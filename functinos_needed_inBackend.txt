A. Invoice Lifecycle System
1. Create Invoice (Draft)

Create invoice record with:

invoice_number (custom allowed)

client_id

invoice_date

due_date

status = Draft

Add line items:

description

quantity

rate

amount

Calculate:

subtotal

GST

total

Allow full editing while status = Draft

2. Serial Number Handling

Backend provides:
GET last_invoice_number

Backend validates uniqueness of invoice_number on create

Backend does NOT auto-generate serial

Invoice number immutable after status != Draft

3. Update Draft Invoice

Allowed only if status == Draft

Can:

change invoice number

edit line items

modify rates

Recalculate totals on each update

4. Generate Invoice

Change status Draft → Generated

Lock invoice (no structural edits allowed)

Freeze totals

Write ledger entry:
Debit: Client Receivable
Credit: Revenue

Store generated timestamp

5. Mark as Sent

Change status → Sent

No financial impact

6. Cancel Invoice (Admin Only)

Only if not fully paid

Change status → Cancelled

Write reversing ledger entry

Keep invoice record (no deletion)

3️⃣ SYSTEM-WIDE RULES

These are non-negotiable logic rules.

1. Financial Integrity

All invoice + payment operations must run inside DB transaction.

No partial invoice creation.

No partial payment application.

2. Decimal Precision

All money values stored as NUMERIC in PostgreSQL.

Never use JS floating arithmetic.

If calculating in JS, use decimal library.

3. Immutability Rules

Cannot:

Edit generated invoice line items

Edit invoice number after generation

Delete transactions

Delete payments

Delete invoices (only cancel)

4. Role Enforcement

Admin:

Cancel invoice

Reverse payment

Modify settings

Operator:

Create invoice

Edit draft

Add payment

Viewer:

Read-only

5. Concurrency Control

For:

Transfers

Bulk billing

Payment application

Use:
BEGIN
SELECT FOR UPDATE
COMMIT

Return 409 on conflict.
