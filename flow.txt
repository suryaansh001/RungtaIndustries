STEP 2 — MODELS (SQLAlchemy ORM)
════════════════════════════════════════════════════════════════════════════════

CONTEXT FOR AGENT:
The schema is v2.0 Ledger-First. Key rules:
  - Master tables store identity ONLY
  - Ledger (transactions table) stores truth
  - NO stored computed columns anywhere:
      coils:        NO holding_days, current_weight_kg, coil_balance, status
      packets:      NO current_balance, status
      transactions: NO coil_balance_after
  - All ENUMs — no free-text for any status/type field
  - processing_stage on coils is ENUM (proc_stage), not VARCHAR
  - rate_unit on pricing_config is ENUM (rate_unit), not VARCHAR
  - transfer status is ENUM (transfer_status), not VARCHAR
  - pricing_config has UNIQUE on (product_type, activity_type, jw_line, effective_from)
  - transfer_orders has CHECK: from_party_id != to_party_id
  - transactions has CHECK: net_weight_kg != 0
  - net_weight_kg sign: +ve = inflow, -ve = outflow

════════════════════════════════════════════════════════════════════════════════
STEP 9 — TRANSFER SERVICE (MOST CRITICAL STEP)
════════════════════════════════════════════════════════════════════════════════

CONTEXT FOR AGENT:
This is the most important business logic in the system.
The transfer service MUST use SELECT FOR UPDATE NOWAIT to prevent two operators
from transferring the same coil simultaneously.
All 8 steps inside the service must run in a single database transaction.
If ANY step fails, the ENTIRE transfer must rollback (no partial state).

PROMPT:
───────
Implement the transfer service in app/services/transfer_service.py and
router in app/routers/transfers.py.

app/services/transfer_service.py:

  execute_coil_transfer(db, data: TransferCreate, user_id) → TransferOrder:

    This function must run entirely inside a single DB transaction.
    Steps:

    STEP A — Validate parties:
      - from_party and to_party must both exist and be active
      - from_party_id must != to_party_id (also enforced by DB CHECK)
      - Raise 404 if either not found, 400 if same party

    STEP B — Lock coils (CRITICAL — prevents race condition):
      SELECT id, coil_number, current_party_id, net_weight_kg,
             size, processing_stage
      FROM coils
      WHERE id = ANY(:coil_ids)
      FOR UPDATE NOWAIT
      -- NOWAIT: if another transaction holds the lock, raise immediately
      -- Do NOT use FOR UPDATE (without NOWAIT) — silent queue = silent corruption

      If lock fails → raise 409 "One or more coils are being modified.
                                  Please retry in a moment."

    STEP C — Validate ownership:
      For EVERY locked coil:
        - coil.current_party_id must == data.from_party_id
        - Raise 400 with specific coil_number if any coil doesn't belong to from_party
      This prevents transferring coils you don't own.

    STEP D — Create TransferOrder:
      INSERT transfer_orders row with status=pending

    STEP E — Create TransferLineItems:
      For each coil, INSERT transfer_line_items with:
        - coil_number_snapshot = coil.coil_number   (frozen NOW)
        - weight_kg_snapshot   = coil.net_weight_kg  (frozen NOW)
        - size_snapshot        = coil.size           (frozen NOW)
        - snapshot_processing_stage = coil.processing_stage (frozen NOW)
        - prev_party_id = from_party_id
        - new_party_id  = to_party_id

    STEP F — Update coil ownership (the actual transfer):
      UPDATE coils SET current_party_id = to_party_id
      WHERE id IN (coil_ids)
      This is the one and only mutation.

    STEP G — Write ledger (both sides):
      For EACH coil write TWO transaction rows:
        Row 1: activity=Coil-Transfer-Out, party_id=from_party_id,
               net_weight_kg = -coil.net_weight_kg  (NEGATIVE)
        Row 2: activity=Coil-Transfer-In,  party_id=to_party_id,
               net_weight_kg = +coil.net_weight_kg  (POSITIVE)

    STEP H — Mark order complete:
      UPDATE transfer_orders SET status=completed

    On any exception → db.rollback() and re-raise
    On success → db.commit()

  reverse_transfer(db, transfer_order_id, user_id) → TransferOrder:
    - Check is_reversible == True on the order, else raise 400
    - Check status == completed, else raise 400
    - Re-run execute_coil_transfer in REVERSE (to→from, same coils)
    - Mark original order status = reversed

Endpoints in app/routers/transfers.py:

  POST   /api/v1/transfers
    Body: TransferCreate
    Calls: execute_coil_transfer
    Returns: TransferResponse (201)
    On lock conflict → 409 with retry message

  GET    /api/v1/transfers
    Query params: from_party_id?, to_party_id?, status?,
                  from_date?, to_date?, page, size
    Returns: PaginatedResponse[TransferResponse]

  GET    /api/v1/transfers/{transfer_id}
    Returns: TransferResponse with line_items nested

  GET    /api/v1/transfers/history
    Queries vw_transfer_history view
    Query params: party_id?, coil_number?, from_date?, to_date?, page, size
    Returns full audit trail

  POST   /api/v1/transfers/{transfer_id}/reverse
    Calls: reverse_transfer
    Requires: admin role
    Returns: new reversed TransferResponse

REFERENCE — KEY BUSINESS RULES (never violate these)
════════════════════════════════════════════════════════════════════════════════

1. LEDGER IS IMMUTABLE
   Never UPDATE or DELETE a transaction row.
   Corrections = new reversal rows with opposite sign.

2. NET_WEIGHT_KG SIGN CONVENTION
   +ve = inflow  (Coil-In, Pkt-In, Transfer-In)
   -ve = outflow (Dispatch, Transfer-Out, Handling, Processing output)

3. RATES FREEZE AT INSERT TIME
   rate_applied and amount_charged on transactions must be set at insert time.
   Never recalculate retroactively when pricing_config changes.

4. COIL WEIGHT IS IMMUTABLE
   coil.net_weight_kg is the original entry weight. Never mutate it.
   remaining_weight = net_weight_kg - SUM(output_weight_kg + scrap_weight_kg)

5. TRANSFERS USE SELECT FOR UPDATE NOWAIT
   Never skip this. Without it, two operators can double-transfer.
   NOWAIT = immediate error. Not FOR UPDATE (which silently queues).

6. TRANSFER SNAPSHOTS ARE IMMUTABLE
   transfer_line_items snapshot fields are written once and never changed.
   They represent state AT THE MOMENT OF TRANSFER.

7. STATUS IS NEVER STORED
   Coil status, packet status, balance — always computed from ledger.
   If you store status physically, it will eventually lie.

8. ORIGINAL_PARTY_ID IS IMMUTABLE
   Set once at creation. Never update it.
   It always answers: "who originally brought this in?"

9. PRICING UNIQUE CONSTRAINT
   Never allow two active rates for the same product+activity+line combo.
   When adding a new rate: auto-close the old one first.

10. ATOMIC TRANSFERS
    All 8 steps of execute_coil_transfer run in ONE transaction.
    Either all succeed or nothing changes. No partial state.
